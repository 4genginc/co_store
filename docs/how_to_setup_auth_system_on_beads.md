# Beads Auth System — Build Procedure

A practice guide for rebuilding the **Auth System** demo end-to-end with Beads
and Claude Code. This document mirrors the actual run that produced this repo:

```
Auth System (epic)
├── Design UI         — ASCII wireframes
├── Backend           — Next.js 16 + Prisma + argon2
│   └── (depends on Design UI)
├── Tests             — vitest suite
│   └── (depends on Backend)
└── Auth Pages        — Browser pages wired to the API
    └── (depends on Design UI)
```

The bead-id prefix (`my_beads-lfn` here) is generated by `bd init` and differs
per repo, so every step below uses **shell variables** instead of hard-coded
IDs. Set them once, reuse them everywhere.

---

## 0. Prerequisites

- `bd` (Beads) installed and on `$PATH`
- Node 20+ and npm 10+
- A clean git repo (`git init && git commit` an initial state)
- Claude Code (or any agent that can take "Work on bead X only" prompts)

---

## 1. Initialize Beads

```bash
bd init
bd setup claude        # writes hooks + CLAUDE.md guidance for Claude Code
```

`bd init` prints the bead-id prefix this repo will use. You don't need to
memorize it — every later step captures IDs into variables.

---

## 2. Create the epic and child beads, capturing IDs into shell variables

`bd create` prints a line like:

```
✓ Created issue: my_beads-lfn — Auth System
```

We can grep that with `awk` and bind the ID to a shell variable. Each block
below creates one bead and stores its ID.

### 2a. Helper

Keep this in the same shell session as the rest of step 2.

```bash
# Returns just the bead id from a `bd create` invocation passed via stdin.
bd_id() { awk '/Created issue:/ { print $4; exit }'; }
```

### 2b. Epic

```bash
EPIC_ID=$(bd create \
  --title="Auth System" \
  --type=epic \
  --priority=2 \
  --description="Build a small Next.js 16 auth demo end-to-end: wireframes, backend, tests, browser pages." \
  | tee /dev/tty | bd_id)

echo "EPIC_ID=$EPIC_ID"
```

### 2c. Child beads

```bash
DESIGN_ID=$(bd create \
  --title="Design UI" \
  --type=task --priority=2 \
  --parent="$EPIC_ID" \
  --description="Design auth UI wireframes for a Next.js 16 web app covering login, signup, and password reset only. Output ASCII wireframes only, no code." \
  --acceptance="ASCII wireframes exist for login, signup, and password reset; include fields, buttons, error states, and navigation links; no code scaffolded." \
  | tee /dev/tty | bd_id)

BACKEND_ID=$(bd create \
  --title="Backend" \
  --type=task --priority=2 \
  --parent="$EPIC_ID" \
  --description="Implement the auth backend for a Next.js 16 app using Prisma + SQLite, argon2 password hashing, httpOnly cookie sessions, and password reset tokens. Stub reset email delivery to console." \
  --acceptance="Backend supports signup, login, logout/session handling, password reset request, and password reset confirm; reset tokens are hashed at rest, single-use, and expire after 1 hour; implementation passes npm install, prisma migrate, and TypeScript typecheck." \
  | tee /dev/tty | bd_id)

TESTS_ID=$(bd create \
  --title="Tests" \
  --type=task --priority=2 \
  --parent="$EPIC_ID" \
  --description="Implement tests for the auth backend covering signup, login, logout/session behavior, password reset request, password reset confirm, reset token expiration/single-use, and session invalidation after reset." \
  --acceptance="Tests cover the required auth flows; npm test passes; npx tsc --noEmit passes." \
  | tee /dev/tty | bd_id)

PAGES_ID=$(bd create \
  --title="Auth Pages" \
  --type=task --priority=2 \
  --parent="$EPIC_ID" \
  --description="Build browser UI pages for the auth system using the existing ASCII wireframes and existing auth API routes. Implement login, signup, logout/session display, password reset request, and password reset confirm pages. Use simple, clean Next.js 16 App Router pages." \
  --acceptance="Users can sign up, log in, view current session, log out, request a password reset, and submit a new password through browser pages; pages are wired to existing API routes; npm test and npx tsc --noEmit pass." \
  | tee /dev/tty | bd_id)
```

### 2d. Persist the variables (optional but recommended)

Shell vars die with the shell. To survive across sessions:

```bash
cat > .beads-ids.env <<EOF
EPIC_ID=$EPIC_ID
DESIGN_ID=$DESIGN_ID
BACKEND_ID=$BACKEND_ID
TESTS_ID=$TESTS_ID
PAGES_ID=$PAGES_ID
EOF
```

Reload later with:

```bash
set -a && source .beads-ids.env && set +a
```

> Tip: add `.beads-ids.env` to `.gitignore` so per-machine bead IDs don't end up
> in version control.

---

## 3. Wire up the dependency graph

```bash
bd dep add "$BACKEND_ID" "$DESIGN_ID"   # Backend depends on Design UI
bd dep add "$TESTS_ID"   "$BACKEND_ID"  # Tests depends on Backend
bd dep add "$PAGES_ID"   "$DESIGN_ID"   # Auth Pages depends on Design UI

bd blocked
bd ready
```

`bd ready` should now show **only** Design UI as ready to work; the others are
blocked behind dependencies.

---

## 4. Execute each bead with Claude Code

Beads define **what** to do; Claude Code does the work. For each bead, claim
it, hand the agent a tightly scoped prompt, then close on acceptance.

```bash
# Generic claim/close idioms used in every bead below
bd update "$ID" --claim
bd close  "$ID" --reason="…"
```

### 4a. Design UI

Prompt template (substitute `$DESIGN_ID` before pasting into Claude Code):

```text
Work on bead $DESIGN_ID only.
Do not touch Backend or Tests.
Do not write backend code.
Do not scaffold the full app.
```

Acceptance: ASCII wireframes for login, signup, and password reset (request +
confirm) live in `docs/auth_ui_wireframes.md`, including fields, buttons, error
states, and nav links.

```bash
bd close "$DESIGN_ID" --reason="ASCII wireframes delivered."
```

### 4b. Backend

```text
Work on bead $BACKEND_ID only.
Use the existing ASCII wireframes in docs/auth_ui_wireframes.md as reference.
Build backend only — no UI pages, no tests yet.
```

Quality gates run by the agent:

```bash
npm install
DATABASE_URL="file:./dev.db" npx prisma migrate dev --name init
npx tsc --noEmit
```

> The `prisma migrate dev` command auto-creates a `.env` file with
> `DATABASE_URL=…` if one doesn't exist. Keep that file — `npm run dev` reads
> it. **Don't commit it.**

```bash
bd close "$BACKEND_ID" --reason="Backend done; npm install / prisma migrate / tsc all pass."
```

### 4c. Tests

```text
Work on bead $TESTS_ID only.
Do not rewrite backend code; only add tests and supporting test infra.
```

Quality gates:

```bash
npm test
npx tsc --noEmit
```

```bash
bd close "$TESTS_ID" --reason="vitest suite green; tsc clean."
```

### 4d. Auth Pages

```text
Work on bead $PAGES_ID only.

Use the existing ASCII wireframes in docs/auth_ui_wireframes.md.
Use the existing API routes in src/app/api/auth/*.

Build browser UI pages only.
Do not rewrite backend auth logic.
Do not rewrite existing tests unless needed for compatibility.

Required pages:
- login
- signup
- password reset request
- password reset confirm
- simple account/session page
- logout action

After finishing, verify:
- user can click through auth flow in browser
- npm test passes
- npx tsc --noEmit passes
```

Browser verification (do this before closing — tests do **not** replace it):

```bash
DATABASE_URL="file:./dev.db" npm run dev
# open http://localhost:3000/
#   sign up -> account view
#   sign out -> landing
#   sign in -> account view
#   /reset  -> enter email -> read the dev console for the reset link
#   open the link -> set new password -> redirect to /login
#   sign in with the new password
```

```bash
bd close "$PAGES_ID" --reason="Browser flow verified end-to-end."
```

---

## 5. Close the epic

When every child is closed:

```bash
bd show "$EPIC_ID"     # confirm 4/4 children complete
bd close "$EPIC_ID" --reason="All child beads complete."
```

---

## 6. Push your work

```bash
git pull --rebase
bd dolt push           # pushes bead history if a Dolt remote is configured
git push
git status             # must say "up to date with origin"
```

## Lessons from this run (worth applying next time)

1. **Decompose the *full* user-visible slice up front.** The first plan only
   had Design UI / Backend / Tests — no UI pages — so the Auth System wasn't
   browser-testable until a fourth bead (`Auth Pages`) was added. Either
   include UI in the initial decomposition, or be explicit that the epic's
   first milestone is API-only.
2. **Wireframes earn their keep when they constrain code.** The Design UI
   bead's ASCII reset link (`/reset?token=…`) became the exact URL the mailer
   stub emits and the page reads — no second guessing.
3. **Browser-test before closing UI beads.** A run that passes `npm test` and
   `tsc` can still 500 in the browser the first time you click a button (a
   real Turbopack/native-module timing case we hit). Click through the flow.
4. **Catch blocks must log.** A generic `Internal server error` with no
   server-side log makes a transient bug nearly impossible to diagnose. Always
   `console.error(err)` in the catch-all of a route handler.
5. **`bd create` IDs into variables, not into prose.** Hard-coded prefixes
   (`test_beads_1-2p1` etc.) rot the moment you `bd init` somewhere else.

## Reference: bead tree

| Variable      | Title       | Type | Depends on    |
| ------------- | ----------- | ---- | ------------- |
| `$EPIC_ID`    | Auth System | epic | —             |
| `$DESIGN_ID`  | Design UI   | task | —             |
| `$BACKEND_ID` | Backend     | task | `$DESIGN_ID`  |
| `$TESTS_ID`   | Tests       | task | `$BACKEND_ID` |
| `$PAGES_ID`   | Auth Pages  | task | `$DESIGN_ID`  |

## Key idea

```text
Epic                = project container
Child beads         = work units
Dependencies        = execution order
Acceptance criteria = close condition
Shell variables     = portable IDs, written once, reused everywhere
```
